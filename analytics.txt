Ключевые задачи:
1) Прием параметров которые вводит пользователь: 
•	параметры груза(вес, габариты, тип)
•	город отправки и доставки
•	желаемая дата доставки
•	приоритет
2) Хранение динамической базы данных перевозчиков и их условий.
Система должна знать:
•	какие есть компании
•	какие у этих компаний рейтинги
•	какая у них специализация(мелкие посылки, сборные грузы, негабарит)
•	какие у них тарифы
•	ограничения
•	сроки 
•	особенности
3) На основе параметров груза система должна получать из базы данных нужные варианты и сравнивать их между собой.
4) Не просто список цен, а сортировка по выбранным критериям и по приоритету который указывает пользователь: «минимальная стоимость», «максимальная скорость» или «наивысшая надежность»
5) Сбор обратной связи.
После завершения доставки пользователь может оценить, насколько реальность совпала с обещаниями (фактические сроки, сохранность груза, качество коммуникации с перевозчиком).
А так же с помощью этой обратной связи корректировать нашу базу данных (рейтинги компаний), а в алгоритмы сравнения вносятся поправки, повышающие точность будущих прогнозов и рекомендаций.

Что делать в первую очередь.
1.	Форма ввода параметров груза
2.	Список перевозчиков
3.	Таблица сравнения предложений
4.	Фильтрация и сортировка по цене, сроку, надежности
5.	Простая система рекомендаций по весам критериев ( типо приоритет «дешево» -> больший вес у цены. Приоритет «быстро» -> больший вес у срока доставки. Приоритет «надежно» -> больший вес у рейтинга и как то это расчитать.
6.	Страница с деталями перевозчика
7.	Хранение данных в базе
8.	Базовая админка или способ редактировать данные
Что делать во вторую очередь.
1.	Подключение реальных API перевозчиков
2.	Сохранение истории запросов
3.	Аккаунты пользователей
4.	Отзывы и оценки после доставки
5.	Расширенная аналитика по направлениям
6.	Уведомление о статусе заявки
7.	Экспорт результатов

«Килл фича, мб что то по типу чтобы система не просто показывала того кого порекомендовала, но еще и объяснила этот выбор, разложив по полочкам в чем плюсы и минусы именно этой компании или придумать какой нибудь отдельный, дополнительный рейтинг чтобы при выборе система анализировала более точно.»

Стек разработчики в основном решайте сами, но просто как варианты найденные в интернете это:
Если на JavaScript/TypeScript:
1. Фронтенд:
•	React или Next.js
•	TypeScript
•	Tailwind CSS или обычный CSS
•	UI-kit по желанию, например shadcn/ui
2. Бэкенд:
•	Node.js
•	NestJS или Express
3. База данных:
•	PostgreSQL
4. ORM:
•	Prisma
5. Авторизация:
•	JWT
6. Документация API:
•	Swagger / OpenAPI
7. Деплой:
•	Docker
•	GitHub
•	при необходимости Nginx
Если на Python:
1. Фронтенд:
•	React / Next.js
2. Бэкенд:
•	FastAPI
3. База:
•	PostgreSQL
4. ORM:
•	SQLAlchemy
5. Авторизация:
•	JWT

Чем пользоваться не нужно:
•	Микросервисы (нужно делать один бекенд)
•	Kafka (маштаб обработки данных намного больше чем требуется в проекте)
•	Сложный ML (лучше сделать простую формулу рейтинга, чем использовать модели обучения, прогнозирования, нейросети и т.д.
•	Собственный парсер всех логистических сайтов (заходить самостоятельно на сайты и вытаскивать от туда цены и данные будет очень муторно и долго. Легче использовать API(если есть) или забить данные вручную в базу)
•	Распределенные кэши(опять же для нашего маштаба обычной базы данных должно хватить)
•	Блокчейн
•	Суперсложную архитектуру

Завайбкодить можно:
•	Страницы интерфейса
•	Формы
•	Таблицы сравнения
•	Карточки перевозчиков(блок с инфой)
•	Админку
•	Шаблонные операции ( по типу добавить перевозчика, удалить или изменить)
•	Фильтры и сортировки (типо показать только дешевле 1000)
•	Простые модальные окна
•	Базовую верстку
Завайбкодить нельзя:
•	Логику расчета рекомендаций( как система выбирает лучший вариант)
•	Формулу ранжирования
•	Структуру базы данных
•	Авторизацию
•	Обработку ошибок
•	Бизнес логику сравнений ( как именно сравниваем компании например: учитываются ли задержки или отзывы и тд)
